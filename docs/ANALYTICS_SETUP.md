# 週次アナリティクス・ダイジェスト セットアップ手順（オーナー専用）

このドキュメントは、**あなた（オーナー）だけ**が受け取る「週次アナリティクス・
ダイジェスト」を有効にするための、一度きりのセットアップ手順です。エンジニアで
なくても進められるよう、画面のメニュー名までそのまま書いています。上から順に進め
てください。

- できあがるもの: 毎週月曜の朝に、先週（月〜日）と前週を比べた **サイトのアクセス
  状況（GA4）と検索の状況（Search Console）の要約**が、LINE または PC のログに届く
  ようになります。
- 数字は**公開されません**。数値は非公開の保管場所（Supabase の private バケット）と
  あなたの PC の中だけに保存されます。GitHub には鍵も数字も一切コミットされません。
- **設定が終わるまでは何も起きません**（スクリプトは安全に「何もせず正常終了」します）。
  なので、途中まで進めて中断しても壊れません。

> 用語ミニ解説
> - **GA4** = Google アナリティクス（サイトに何人来たか等を見るツール）。
> - **Search Console** = Google 検索での表示回数・クリック数・掲載順位を見るツール。
> - **サービスアカウント** = 「閲覧専用のロボット用アカウント」。あなたの Google
>   ログイン（メール＋パスワード）を渡すのではなく、**閲覧だけできるロボットに、
>   自分のデータを『見せてあげる』**イメージです。パスワード共有ではないので安全です。

---

## 0. 全体の流れ（チェックリスト）

1. [ ] **Google Cloud** でプロジェクトを作り、必要な API を有効化し、**サービス
       アカウント（閲覧ロボット）**を作って **JSON の鍵ファイル**を1つダウンロードする
2. [ ] **GA4** で、そのロボットのメールアドレスを「**閲覧者**」として追加する
       （＋ データ保持を14ヶ月に変更 ＋ 自宅アクセスを除外）
3. [ ] **Search Console** で、同じメールアドレスをユーザー追加する
4. [ ] ダウンロードした鍵を `secrets\ga-service-account.json` に置き、`.env.local` に
       `GA4_PROPERTY_ID` を書き足す
5. [ ] `python scripts/analytics_weekly_report.py --dry-run` で動作確認する
6. [ ] （最後に）オーナーが Task Scheduler に毎週月曜 09:00 の定期実行を登録する
       （コマンドは本書の最後にあります）

所要時間の目安: 20〜30分。1〜4は一度だけやれば OK です。

## 1. Google Cloud でロボット（サービスアカウント）と鍵を用意する

ブラウザで <https://console.cloud.google.com/> を開き、あなたの Google アカウントで
ログインします。

### 1-1. プロジェクトを作る

1. 画面上部の**プロジェクト選択メニュー**（左上、Google Cloud ロゴの右にある
   ドロップダウン）をクリック。
2. 右上の「**新しいプロジェクト**」をクリック。
3. 「**プロジェクト名**」に分かりやすい名前（例: `meguribi-analytics`）を入力し、
   「**作成**」をクリック。
4. 作成後、もう一度上部のプロジェクト選択メニューで、**今作ったプロジェクトが
   選ばれている**ことを確認します（以降の作業はこのプロジェクトの中で行います）。

### 1-2. 必要な API を2つ有効化する

1. 左上の**ナビゲーションメニュー**（≡ 三本線アイコン）→「**API とサービス**」→
   「**ライブラリ**」をクリック。
2. 検索窓に `Google Analytics Data API` と入力して開き、「**有効にする**」を
   クリック。
3. 続けて同じ「ライブラリ」で `Google Search Console API` を検索して開き、
   「**有効にする**」をクリック。

（この2つが、ロボットが GA4 と Search Console の数字を読むための入口です。）

### 1-3. サービスアカウント（閲覧ロボット）を作る

1. 左上のナビゲーションメニュー（≡）→「**IAM と管理**」→「**サービス アカウント**」。
2. 上部の「**＋ サービス アカウントを作成**」をクリック。
3. 「**サービス アカウント名**」に例えば `meguribi-report-bot` と入力（ID は自動で
   埋まります）→「**作成して続行**」。
4. 「**このサービス アカウントにプロジェクトへのアクセスを許可する**」（ロールの
   付与）は**何も選ばずに**「**続行**」。
   - 理由: このロボットに必要なのは「GA4 と Search Console を見る権限」だけで、
     それは後の手順（2・3）で GA4／Search Console 側から直接与えます。ここで
     プロジェクト全体のロールを与える必要はありません（最小権限）。
5. 「**ユーザーにこのサービス アカウントへのアクセスを許可**」も**空欄のまま**
   「**完了**」。

### 1-4. JSON の鍵ファイルをダウンロードする

1. 作成されたサービスアカウントの一覧で、今作ったものの**メールアドレス**
   （`meguribi-report-bot@<プロジェクトID>.iam.gserviceaccount.com` のような形）を
   クリック。
   - この**メールアドレスは後で2回使う**ので、メモ帳などにコピーしておいてください。
2. 上部の「**キー**」タブをクリック。
3. 「**鍵を追加**」→「**新しい鍵を作成**」。
4. キーのタイプで「**JSON**」を選び、「**作成**」。
5. **JSON ファイルが自動でダウンロード**されます（例: `meguribi-analytics-xxxx.json`）。
   このファイルが「ロボットの秘密の鍵」です。**絶対に他人に渡さない／どこにも
   アップロードしない**でください。置き場所は手順4で指定します。

> ⚠️ このファイルは再ダウンロードできません（無くしたら鍵を作り直し）。手順4で
> 置き場所に移すまで、ダウンロードフォルダから消さないでください。

## 2. GA4 側の設定

ブラウザで <https://analytics.google.com/> を開きます。**対象のプロパティ
（めぐりび）**が選ばれていることを確認してください。

### 2-1. ロボットを「閲覧者」で追加する

1. 左下の「**管理**」（歯車アイコン ⚙）をクリック。
2. 右側の「**プロパティ**」列で「**プロパティのアクセス管理**」をクリック。
3. 右上の「**＋**」→「**ユーザーを追加**」。
4. 「**メールアドレス**」に、手順1-4でメモした**サービスアカウントのメールアドレス**
   （`...iam.gserviceaccount.com`）を貼り付け。
5. 「**新規ユーザーにメールで通知する**」の**チェックを外す**（相手はロボットで、
   メール受信箱が無いため）。
6. 「**直接的な役割とデータの制限**」で「**閲覧者**」を選択。
7. 右上の「**追加**」をクリック。

### 2-2. プロパティ ID を確認する（`.env.local` に書く番号）

1. 「**管理**」→「**プロパティ**」列の「**プロパティの詳細**」（または「プロパティ
   設定」）を開く。
2. 画面右上あたりに「**プロパティ ID**」という**数字（9桁前後）**が表示されます。
   例: `493218765`。この番号を控えておきます（手順4で使います）。
   - ※「G-XXXXXXXXXX」（測定 ID）とは**別物**です。使うのは**数字だけの
     プロパティ ID** の方です。

### 2-3. データ保持期間を「2ヶ月 → 14ヶ月」に変更する

初期設定のままだと**2ヶ月より前のデータが消えて前週比が取れなくなる**ため、必ず
14ヶ月に伸ばします。

1. 「**管理**」→「**プロパティ**」列の「**データの設定**」→「**データ保持**」。
2. 「**ユーザーデータとイベントデータの保持**」の中の「**イベントデータの保持**」を
   「**2 か月**」から「**14 か月**」に変更。
3. 「**保存**」をクリック。

### 2-4. 自宅からのアクセスを除外する（内部トラフィック）

自分でサイトを確認したアクセスが数字に混ざらないよう、自宅の IP を「内部
トラフィック」として除外します。**（A）自宅 IP のルール作成**と**（B）フィルタの
有効化**の2段階です。

まず自宅のグローバル IP アドレスを調べます: ブラウザで `確認くん` または
`what is my ip` を検索すると、`203.0.113.45` のような数字が表示されます。これを
控えておきます。

**（A）内部トラフィックのルールを作る**

1. 「**管理**」→「**プロパティ**」列の「**データの収集と修正**」→「**データ
   ストリーム**」。
2. 対象の**ウェブ ストリーム**（めぐりびのサイト）をクリック。
3. 「**Google タグ**」欄の「**タグの設定を行う**」をクリック（または「**タグ設定を
   構成**」）。
4. 「**設定**」セクションの中の「**内部トラフィックの定義**」をクリック。
5. 「**作成**」をクリック。
6. 「**ルール名**」に例えば `自宅` と入力。`traffic_type` の値は既定の `internal`
   のままで OK。
7. 「**IP アドレスの照合タイプ**」で「**IP アドレスが次と等しい**」を選び、先ほど
   調べた**自宅のグローバル IP**を入力。
8. 右上の「**作成**」をクリック。

**（B）データフィルタを「有効」にする**

1. 「**管理**」→「**プロパティ**」列の「**データの設定**」→「**データフィルタ**」。
2. 「**Internal Traffic**」という名前のフィルタをクリック。
3. 「**フィルタの状態**」を「**テスト**」から「**有効**」に変更。
4. 「**保存**」をクリック（確認ダイアログが出たら承認）。

> これで、自宅からの閲覧はレポートの数字から除外されます。

## 3. Search Console 側の設定

ブラウザで <https://search.google.com/search-console> を開きます。

### 3-1. ロボットをユーザー追加する

1. 左メニュー最下部の「**設定**」（歯車）をクリック。
2. 「**ユーザーと権限**」をクリック。
3. 右上の「**ユーザーを追加**」。
4. 「**メールアドレス**」に、手順1-4の**サービスアカウントのメールアドレス**を貼り付け。
5. 「**権限**」は「**制限付き**」で OK（閲覧だけできれば十分）。
6. 「**追加**」をクリック。

### 3-2. プロパティの形式を確認する（`GSC_SITE_URL` の値）

Search Console のプロパティには2種類あり、スクリプトに教える値が変わります。

- **URL プレフィックス型**: `https://www.meguribi.jp/` のように**フル URL**。
- **ドメイン型**: `meguribi.jp` のように**ドメインだけ**。

**このリポジトリの状況からは「URL プレフィックス型（`https://www.meguribi.jp/`）」で
登録されていると判断しています。** 根拠は次の2つです。

- `frontend/public/googlea5f06853b9c777a0.html` という **HTML ファイル方式の所有権
  確認ファイル**がリポジトリに存在します。この確認方式は **URL プレフィックス型
  でしか使えません**（ドメイン型は DNS の TXT レコードでの確認が必須）。
- サイトの正規 URL（`frontend/src/lib/siteUrl.ts` の本番既定）が
  `https://www.meguribi.jp` です。

そのため、スクリプトの既定値は `https://www.meguribi.jp/`（末尾スラッシュ付き）に
してあり、**通常は何も設定しなくて OK** です。

念のため実際の登録を確認するには:

1. 左上の**プロパティ選択メニュー**（ドロップダウン）を開く。
2. 一覧で `https://www.meguribi.jp/` のように**フル URL で表示**されていれば
   URL プレフィックス型です（そのまま既定でOK）。
3. もし `meguribi.jp` のように**ドメインだけ**で表示されていたら**ドメイン型**なので、
   手順4で `.env.local` に次の1行を足してください（`sc-domain:` を付けるのがルール）:

   ```
   GSC_SITE_URL=sc-domain:meguribi.jp
   ```

## 4. 鍵ファイルの置き場所と `.env.local` の設定

### 4-1. 鍵ファイルを置く

1. リポジトリ直下（`...\riental-lounge-monitor-main\`）に **`secrets` という名前の
   フォルダ**を新しく作ります。
2. 手順1-4でダウンロードした JSON ファイルを、この `secrets` フォルダに移動し、
   **ファイル名を `ga-service-account.json` に変更**します。
3. 最終的なパスはこうなります:

   ```
   ...\riental-lounge-monitor-main\secrets\ga-service-account.json
   ```

> ✅ この `secrets\` フォルダは `.gitignore` 済みなので、**GitHub には絶対に
> アップロードされません**。安心して鍵を置いてください。

### 4-2. `.env.local` にプロパティ ID を書き足す

リポジトリ直下の **`.env.local`**（無ければ新規作成。これも `.gitignore` 済み）を
テキストエディタで開き、手順2-2で控えた**プロパティ ID の数字**を書き足します:

```
GA4_PROPERTY_ID=493218765
```

（数字はあなたのプロパティ ID に置き換えてください。）

- Search Console が**ドメイン型**だった場合のみ、手順3-2の `GSC_SITE_URL=...` の行も
  追加します。URL プレフィックス型なら不要です。
- LINE で受け取りたい場合は、`LINE_USER_ID`（自分の LINE ユーザー ID）も設定します
  → 詳細は後述の「LINE で受け取る」を参照。

### 4-3. 必要なライブラリを入れる（初回のみ）

コマンドプロンプト（または PowerShell）でリポジトリ直下に移動し、次を実行します:

```
pip install -r requirements.txt
```

（このスクリプトは新しく `google-auth` を1つだけ使います。上記コマンドで一緒に
入ります。）

## 5. 動作確認（`--dry-run`）

設定が正しいか、**保存も送信もしない安全モード**で確認します。リポジトリ直下で:

```
python scripts/analytics_weekly_report.py --dry-run
```

- **まだ設定できていない場合**の出力（安全に何もせず終了します。エラーではありません）:

  ```
  [analytics] 週次アナリティクスの認証情報が見つかりません（未セットアップ）。
    このスクリプトは設定が完了するまで安全に何もしません（正常終了）。
    有効化の手順は docs/ANALYTICS_SETUP.md を参照してください。
    必要なもの: GA4_PROPERTY_ID（.env.local）と secrets/ga-service-account.json（鍵ファイル）。
  ```

- **設定が完了している場合**の出力（先週の要約が画面に表示されます。まだ保存も
  LINE 送信もしません）:

  ```
  [analytics] 対象週 2026-07-06〜2026-07-12（前週 2026-06-29〜2026-07-05）を集計します。

  ===== DRY RUN: 生成したダイジェスト =====

  【めぐりび 週次アナリティクス】
  対象: 2026-07-06〜2026-07-12
  （前週比: 2026-06-29〜2026-07-05）

  ■ サイト全体（GA4）
  ・アクティブユーザー: 1,234（前週比 +12.2% ↑）
  ・セッション: 1,500（前週比 +7.1% ↑）
  ・ページビュー: 4,200（前週比 +7.7% ↑）

  ■ 検索流入（Search Console）
  ・クリック: 320（前週比 +14.3% ↑）
  … （以下、人気ページ・検索クエリ・イベント利用状況が続きます）
  ```

  ここまで表示されれば成功です。`--dry-run` を外して実行すると、生データ JSON を
  非公開バケットへ保存し、ダイジェストを PC のログ（`%TEMP%` フォルダ）に書き出し、
  `LINE_USER_ID` があれば LINE にも送ります。

> 補足: Search Console のデータは**直近2日ほど集計が確定していない**ことがあります。
> 月曜の朝に「先週の日曜」を含めて集計するため、土日の検索データが少なめに出る場合が
> ありますが、これは Google 側の仕様です（ダイジェスト末尾にも注記が入ります）。

## 6. LINE で受け取る（任意）

このスクリプトは、既存の相席屋アラートや PAT 期限通知と**同じ LINE Push の仕組み**を
使います。使う環境変数は次の2つで、どちらも `.env.local` に入れます。

- `LINE_CHANNEL_ACCESS_TOKEN` … LINE の送信トークン（**既に設定済み**のはずです）。
- `LINE_USER_ID` … **送り先（あなた自身の LINE ユーザー ID、または受信グループ ID）**。
  現状これは未設定なので、**設定するまで LINE 送信は行われません**（その場合でも
  ダイジェストは PC のログに保存され、処理は正常終了します）。

`LINE_USER_ID` が分からない場合は、GitHub Actions 用にも同じ `LINE_USER_ID` を
使っています（`.github/workflows/check-pat-expiry.yml`）。そこに登録済みの値と
同じものを `.env.local` にも書けば、この週次ダイジェストも同じ宛先に届きます。

```
LINE_USER_ID=Uxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

`LINE_USER_ID` を設定しないまま運用しても問題ありません。その場合は毎週
`%TEMP%\analytics_weekly_<週初の月曜>.txt` にダイジェストが保存されます。

## 7. 定期実行の登録（Task Scheduler・最後の仕上げ）

**この登録はオーナー（統括）が最後に行います。** 上の手順1〜5が終わって `--dry-run`
が成功したら、**管理者権限の PowerShell**で次を実行します（既存の
`MEGRIBI-warm-cdn` と同じ流儀＝ `/RU SYSTEM` の「ログオン状態に関わらず実行」）。

```powershell
$py   = "C:\Users\ahos1\AppData\Local\Programs\Python\Python314\python.exe"
$root = "C:\Users\Public\共有データ系\ORIENTAL\ORIENTAL\riental-lounge-monitor-main"

schtasks /Create /TN "MEGRIBI-analytics-weekly" /SC WEEKLY /D MON /ST 09:00 /RU SYSTEM /F `
  /TR "$py $root\scripts\analytics_weekly_report.py"

# PC がスリープ等で 09:00 を逃しても、起動後すぐ実行されるようにする
$s = New-ScheduledTaskSettingsSet -StartWhenAvailable -WakeToRun -AllowStartIfOnBatteries `
     -DontStopIfGoingOnBatteries -ExecutionTimeLimit (New-TimeSpan -Minutes 15)
Set-ScheduledTask -TaskName "MEGRIBI-analytics-weekly" -Settings $s
```

- `/SC WEEKLY /D MON /ST 09:00` = 毎週月曜の 09:00 に実行。
- `/RU SYSTEM` = ログオンしていなくても実行（`MEGRIBI-warm-cdn` と同じ理由。
  詳細は `plan/CDN_WARMING_LOCAL.md`）。
- 手動で今すぐ1回テスト実行するには: `schtasks /Run /TN "MEGRIBI-analytics-weekly"`

## 8. よくあるつまずき

| 症状 | 対処 |
|---|---|
| `--dry-run` が「認証情報が見つかりません」と出る | `secrets\ga-service-account.json` の**ファイル名**と、`.env.local` の `GA4_PROPERTY_ID` の**数字**を再確認。 |
| `google-auth が未インストール` と出る | リポジトリ直下で `pip install -r requirements.txt` を実行。 |
| GA4 の数字が全部 0 | 手順2-1の**閲覧者追加**ができているか、手順2-2の**プロパティ ID**（G- ではなく数字）が合っているか確認。 |
| 検索（Search Console）の数字が全部 0 | 手順3-1のユーザー追加、手順3-2の**プロパティ形式**（URL プレフィックス／ドメイン）と `GSC_SITE_URL` を確認。 |
| 前週比が「新規」ばかり | データ保持が14ヶ月になっていない可能性（手順2-3）。反映には少し時間がかかる場合あり。 |
| LINE に届かない | `.env.local` に `LINE_USER_ID` があるか確認。無ければ PC のログ（`%TEMP%`）に保存されています（正常）。 |
